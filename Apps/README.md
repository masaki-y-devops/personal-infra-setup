個人サーバー構築の記録
====================

Nextcloud
---------

### 経緯

「商用でないクラウドストレージ」に興味がわき、パフォーマンスはどのようなものなのか、どうすればよりセキュアに設定ファイルなどを保存できるのか？を検証したいと思ったのがきっかけです。

まずはVPSで、ベーシックなLAMP環境の構築に挑戦しました。

### 選定理由

基盤はまず最初に無料であることに惹かれ、OCI Always Free Tierから始まり、さくらのVPS、AWS Lightsailの3サービスで構築を体験しました。

プライベートで使用するため、VPSに割り当てられた元々のパブリックIPをVPS外部のパケットフィルター、iptablesで遮断、VPN（tailscale)のみで運用する計画を立て、そのように運用できました。

Nextcloudでは、内蔵のインストールチェッカーがあり、そのエラーをつぶしていくことに注力し、これが基礎的な自走力を養うきっかけになったと考えています。

### 主なトラブルシューティング

#### LAMP環境の基礎的な構成理解の促進。

右も左もわからない状態からVPSを借り、設定を試しましたが、基本操作はArchやNixOSで慣れており、CLI操作に戸惑いはありませんでした。

しかし、Webサービスの背後にWebサーバーサービス、DBサーバーサービスが控えて連携していることをクライアントOSを触っていた時期は意識していなかったので、「なぜ動かないのだろう？」と悩みながら、ドキュメントを頼りに設定を詰めていきました。

#### PHP関連の設定。

それぞれで設定ファイルの位置が異なるなど、PHP設定（FPM,OPCACHEまわり）に苦戦しましたが、公式ドキュメントやエラーメッセージのWeb検索が役立ちました。

FreshRSS
--------

### 経緯

元々ニュースサイトを巡回して読むことが好きで、商用サービス(Feedly）などを使用していましたが、フィード数制限のないセルフホスト型に興味があり導入しました。

### 選定理由

Tiny Tiny RSSという選択肢もありましたが、

- FreshRSSのほうが比較的新しいプロジェクトであること

- インターネット上でのWeb UIを見た感想としてシンプルそうで自分好みだったこと

が決め手となり、試すことにしました。

### 主なトラブルシューティング

同アプリはNextcloudと同様にPHP主体のアプリケーションであることから、Nextcloud構築を経験した後の挑戦だったため、比較的スムーズに導入できたと記憶しております。

しかしながら、FreshRSSを使用していくにつれ、RSSによる情報収集の限界を感じ、後述するchangedetection.ioの構築挑戦への足掛かりになりました。

changedetection.io
------------------

### 経緯

このWebサービスは、RSS配信をそもそもしていない、あるいは取りやめているWebサイト様が多くなってきたことから、常識的な頻度でスクレイピングを試してみたいと考え、構築をしました。

前者2つはベアメタルでの構築でも機能的に不自由しませんでしたが、このアプリは、Docker Compose版の開発がより活発で、pip版はヘッドレスブラウザの対応が不十分・遅かったりしたので、同技術スタックに触れるきっかけになりました。

構築にあたり、Docker（Compose）の知識を再確認することが必要であり、学習に資する構築となりました。

### 選定理由

Distill,VisualPingさんなど商用サービスもありますが、セルフホスト型の自由とシンプルさを兼ね備えていたので導入を決定しました。

構築後、微調整は必要でしたが、登録数の制限等はなく、意図したWeb巡回環境となりました。

### 主なトラブルシューティング

#### Dockerによる抽象化のメリット・デメリットの認識。

ベアメタル構築と異なり、Docker Composeは導入・稼働が比較的楽でしたが、コンテナの部分的な停止・強制終了も多く、ブラックボックス化によるトラブルシューティングの仕方の変化を意識する必要がありました。

#### docker-compose.ymlの編集必要個所の特定。

Docker Compose完全初心者としての設定だったため、「コメントアウト、追記してコンテナ稼働→動かない、エラーで止まる」を繰り返しながら設定を探りました。

最低限必要なものとして、changedetection.io本体のコンテナとスクレイピングのためのヘッドレスブラウザのコンテナが必要なので、その点を意識したのがポイントだったと思います。

#### マシンスペックと運用コストの調整の問題。
3アプリの中でもヘッドレスブラウザを稼働させる必要がある関係上、ある程度のマシンスペックを要求されることがわかりました。

借りているVPSのスペック限界（月額料金の関係でLightsail,メモリ2GBインスタンス）で、コンテナが知らないうちにハングアップしていることが頻繁に発生しました。

しかし、運用コストとの兼ね合いが重要になることから、グレードアップ以外の方法を探り、結果的に定期的にコンテナを立ち上げなおすことで実用上問題ないレベルにまで調整することができました。

#### あえてリバースプロキシでApacheを使ったことでのトラブル。

tailscaleにおけるMagicDNSのSSL証明書を利用するため、changedetection.io内蔵のWebサーバーを背後にして、リバースプロキシとしてApacheを設定しました。

その際に```ProxyPass```,```ProxyPassReverse```の記述の末尾にスラッシュを付けないで設定したことで、コンテンツが正常に読み込まれず何時間もハマった経験をしました。

当時、生成AIにも助けを求めながら、適宜応答を修正・解釈してトラブルシューティングにあたったのを覚えています。

#### メール通知関連の試行錯誤。

変更検知後の通知として、Zoho MailのSMTPサーバーの連携が必要でしたが、ワンライナーで認証を通すことに手間取り、AIのアシストでやっと突破でき、ファクトチェックをしながら適宜AIを活用する意義も学びました。

個人Linuxデスクトップ環境構築スクリプトの記録
=========================================

Arch Linux
----------

### 概要

個人用デスクトップ環境を構築します。

インストール用環境を一時展開した後、インストール先のディスクを最もシンプルな形なパーティションに再構成します。

その後、インストール→ブートローダー再設定→設定済みi3wm＋個人アプリ導入までを行います。

### 使用例
Manjaroライブ環境からinstall.sh実行で自分専用の環境構築を完了するように構成しました。

- USBブートでManjaroライブ環境を立ち上げ、

- インターネット接続を行い、

- ```install.sh```をライブ環境にコピーし、

端末で、

~~~
~~chmod +x ./install.sh
./install.sh~~

cd /path/to/dir ## install.sh保存ディレクトリへ移動
sudo ./install.sh
~~~

を実行して再起動すれば完了します。

**追記：実行にroot権限なため修正。ユーザーへの実行権限付与は任意でいいと思われます。**

### 主なトラブルシューティング

#### 異なるディストロのライブ環境からのインストールにおける、bootstrap一時環境の構築。

Arch Linux開発チームさんは```archinstall```といったインストール簡素化スクリプトを公式で用意してくださっています。

しかし、せっかくだし学習もかねて、どのようなOS構造になっているか知りたいと思い、あえて他のディストロからのインストールをしてみました。

ライブ環境にManjaroを選んでいますが、パッケージに100％の互換性があるとは限らないと考え、bootstrapを用いて使い捨ての一時環境を作っています。

その際に、各種マウントやpacmanの初回処理が特徴的であり、日本語と英語のArchWikiを両方読み、実験用のUMPCの環境を何度もぶっ壊しながら、問題なく動く方法を探りました。

#### 設定ファイル書き出しの際の変数展開の有無の制御。

所々で設定ファイルをヒアドキュメントとして書き出して設定する方法をとっています。

しかし、変数展開させるべきではない場面で変数が空として書き出され構文エラーとなったり、設定エラーが多発したため、適宜バックスラッシュでエスケープしたり、

~~~
cat << 'EOF' | sudo tee ...
~~~

といった形で変数展開を防ぐ必要がありました。

#### ユーザーファイルの権限問題。

i3,i3statusの設定ファイルをほかのシステム設定ファイルと同様に書き込むと、所有者がrootとなり、そのままだとユーザーでログインしたときに設定ファイルが読めません。

気づいた後は「なんだそんなことか」と思えましたが、これに何時間もの間気づけず、試行錯誤をしました。

~~./install.sh~~

cd /path/to/dir ## install.sh保存ディレクトリへ移動
sudo ./install.sh
~~~

を実行して再起動すれば完了します。

**追記：実行にroot権限なため修正。ユーザーへの実行権限付与は任意でいいと思われます。**

### 主なトラブルシューティング

#### Nix言語の書式問題。

まったく初めての書き方で、「結果」から記述していくスタイルは全く初めてで楽しめました。

その反面、関数の選定ミスや記述ミスに加え、コンマの抜けによるコンパイルエラーが多発し、時間を費やした面はあります。

#### Archと比較した初期設定の癖の強さ。

Archよりも、NixOSはデフォルトの設定のクセが強い（明示していない設定は、デフォルトのものが適用解釈される）と感じることもあり、

主観論ではありますが、自分的により「クリーン」と感じる状態にもっていくのがたいへんだったと思います。

#### スクリプトの書き方、実行の流れの問題。

関数型OSという慣れない環境特有のはまりポイントだったかもしれません。

```installer.sh```をヒアドキュメントで書き出し、nixパッケージマネージャーをセットアップしたうえで、それを実行。

大まかな実行内容としては、```configuration.nix```を編集、```nixos-install```を実行するというシンプルな流れだが、これを見つけるのに手間取りました。

愚直に公式サイトやドキュメントをよく読み、組み立てればうまくいきやすいことを学びました。

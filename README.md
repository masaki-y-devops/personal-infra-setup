## はじめに
独学でのPC・サーバー環境構築で詰まったところと試行錯誤、解決のプロセスの記録をまとめました。

私見ではありますが、実務は趣味による構築と違い、チームワークの有無や厳しい仕様の要求、納期等といった強度の制約はなく、その点で全く異なるものであると重く深く受け止めております。

その一方で、実務のための基礎として、趣味で試行錯誤した記録は基礎知識としては有用となるとも考えますので、自習の成果および、それが実務にどう生かせると考えるかを記載しています。

「実務は甘くない」ことを念頭に、いままで自身が経験・自習してきたことをまとめています。

実際に作成したスクリプト、作業手順は各ディレクトリ以下に格納してありますので、ぜひご覧いただければ幸いです。

## 設定を書き始めたきっかけ・なぜこうした活動が楽しいと感じるのか
小学生くらいのころに親のお下がりで古いノートパソコンをもらい、Windowsがスペック上厳しかったので書店でインストールDVD付きの書籍を買ってもらい、Ubuntu（当時は8.04あたり）を触り始めたのが最初のきっかけです。

当時は標準環境でメモやブラウザを動かすことが目的で、初めてWindowsではないOSの可能性に触れました。

大学時代には、秋葉原に足しげく通いました。中古ノートパソコンを買っては色々なディストロをインストールしたり、自作PCを少ない予算の中で組んでLinuxをインストールしていました。このころから「最小構成」にこだわり始めます。

当時のお気に入りのCPUは「Celeron G3900T」でした。このころから「パズルのように組み立てる」ディストロの筆頭ともいえるArchの面白さに触れていきます。

その後、NixOSやGentooに加え、自分でWebサービスを動かしてみたいと思い、サーバー環境構築にも挑戦しました。VPS（さくら、AWS Lightsail、OCI Always Free Tier)にも触れました。

同時に、小さいころに祖父からもらったPDAや電子辞書のお下がりを使うのが好きで、手のひらサイズの情報端末にも幼いながらも憧れがありました。

そのつながりで超小型PC、UMPC好きになりました。GPD製品をはじめとするUMPCで自分の設定が活きると楽しく、それが学習全般のモチベーションになっています。

## 設定探求の原動力・なぜ知識を深堀したいと考えたのか
私の技術習得の原動力は、「理想のコンピューティング環境を、いかに迅速かつ正確に再構築（レプリケーション）できるか」という探求心にあります。

目的達成のために、例えばArch LinuxのEFISTUBによるブート最適化やi3wm,Xmonadタイルマネージャーの活用による軽量化、NixOSによる宣言的な環境管理、xrandr,udevを用いたハードウェア制御、自宅サーバーやVPSを用いた、Tailscale VPN閉域網接続を活用してのNextcloud,FreshRSS,ChangeDetection.ioといったプライベート趣味空間の構築など、低レイヤーからクラウドまで必要な技術を柔軟に取り入れてきました。

特定の抽象化技術に固執することをできるだけ避け、**基本的なLAMP環境といったベーシックな構成を重視し、「課題解決、効率化のために最適な道具を選び、最短距離で形にする」スタンス**を大切にしています。

現在は、断片化している知識を実務を通じて体系化し、より堅牢なインフラ構築に貢献したいと考えています。

## これらがなぜ実務に生かせると考えるのか

**システムをコードとして記述し、それを実行することで同一の個人OS環境を再現できるようにすること**に情熱を注いできました。

→　のちに知った「Infrastructure as a Code」の基礎と共通すると考え、運用属人化の軽減やシステム再現性の強化の有用性に共感しております。

作成したシステム構成は、稚拙かつシンプルではありますが、例えば```archintall```スクリプトやAWSやOCI、さくらのスタートアップ**スクリプトに可能な限り頼らず、できるだけベーシックな構造をスクラッチから構成・構築すること**を心掛けてきました。

→　泥臭いアプローチで一からパーツを組み立てるようにして環境構築した経験は、たとえツールの流行り廃りがあっても、当該知識はより根幹に近く、無駄になりにくい実務における知識となりうると考えております。

## 今後の課題点
### スクリプティングにおいて、「冪等性」「可搬性」の意識が不十分なこと

今までのスクリプト作成にあたっては、個人用途のため、「とりあえず動けばよし」とし、また、エラーや処理失敗を恐れず、経験値を得るため「雑でもとりあえず動かしてみて達成感を得て、細部は後から肉付けする」というポリシーで行ってきました。

しかし、業務上行う開発は、遅延した場合自分が困るに過ぎない個人用途と異なり、それが許されず、お客様に迷惑にならないようにしなければならないのは当然のことと認識しております。

したがって、知識の更なるブラッシュアップが必要不可欠であり、実行結果がブレないようにする冪等性の確保、より応用範囲を広げるための「可搬性」を、これまで以上に意識する必要があると痛感しています。

### サーバー環境構築の自動化の余地があること

サーバー構築自動化のための方法として、Ansible等がありますが、これらを現在勉強中でございます。

自動化ツールの必要性となる背景には、IaCの考え方が根底にあると考え、NixOSを宣言的に構成し、構築してきた経験が、役に立つものと確信しております。

### 使用言語が主にシェルスクリプトやバッチファイル、PowerShellに限定されており、Pythonといった、より自由度・可搬性が高いプログラミング言語の理解が浅いこと

自分の主な関心がOS,ハードウェアの調整寄りだったこともあってか、Windows/Android/iPhone等を対象とするフロントエンドアプリ開発につきましては、ほとんどしてきませんでした。

今後は、より汎用的とされるPythonをはじめ、他の言語にも手を広げる必要性を感じています。

### Git,GitHubによるバージョン管理にまだ慣れていないこと

以前よりスクリプティング・自動化に没頭しておりましたが、「自分のコードは一般に公開できるレベルまで洗練・完成されていない」と考え、公開を控えて、個人のDropboxなどのクラウドストレージにスクリプトを保存・蓄積してまいりました。

しかし、「公開しないならそれは存在していないのと同じ」であると気づき、不完全でも公開して、ほかの人に閲覧いただき、自分の進捗を知っていただくことが有益であると考えを変え、本GitHubページ上に掲載するに至りました。

そのため、GitHubならびにオフラインで動作させるGitも、経験が不足しており、大いに反省すべき点ととらえ、認識しております。

## おわりに

細部は粗削りな部分があることは確かですが、

- OS構築からサーバーインフラの基本的な構築において、基礎的なトラブルシューティングは経験済みであること

- OSやサービスを動かすうえで直面する「わからないこと、詰まったこと」をある程度自分で解決して前へ進める自走力

が現在の自分の強みであると認識しています。

これからも、自分の好奇心とインフラ理解に対する探求心をもって、学習を続けていく所存です。

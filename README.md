## TL;DR
独学でのPC・サーバー環境構築で詰まったところと試行錯誤、解決のプロセスの記録をまとめました。

私見ではありますが、実務は趣味による構築と違い、チームワークの有無や厳しい仕様の要求、納期等といった強度の制約はなく、その点で全く異なるものであると重く深く受け止めております。

その一方で、実務にあたるための基礎として、趣味で試行錯誤した記録は基礎知識としては有用となるとも考えますので、自習の成果および、それが実務にどう生かせると考えるかを記載しています。

「実務は甘くない」ことを念頭に、いままで自身が経験・自習してきたことをまとめています。

実際に作成したスクリプト、作業手順は各ディレクトリ以下に格納してありますので、ぜひご覧いただければ幸いです。

## 設定を書き始めたきっかけ・なぜこうした活動が楽しいと感じるのか
小学生くらいのころに親のお下がりで古いノートパソコンをもらい、Windowsがスペック上厳しかったので書店でインストールDVD付きの書籍を買ってもらい、Ubuntu（当時は8.04あたり）を触り始めたのが最初のきっかけです。

当時は標準環境でメモやブラウザを動かすことが目的で、初めてWindowsではないOSの可能性に触れました。

大学時代には、秋葉原に足しげく通いました。中古ノートパソコンを買っては色々なディストロをインストールしたり、自作PCを少ない予算の中で組んでLinuxをインストールしていました。このころから「最小構成」にこだわり始めます。当時のお気に入りのCPUは「Celeron G3900T」でした。このころから「パズルのように組み立てる」ディストロの筆頭ともいえるArchの面白さに触れていきます。

その後、NixOSやGentooに加え、自分でWebサービスを動かしてみたいと思い、サーバー環境構築にも挑戦しました。VPS（さくら、AWS Lightsail、OCI Always Free Tier)にも触れました。

同時に、小さいころに祖父からもらったPDAや電子辞書のお下がりを使うのが好きで、手のひらサイズの情報端末にも幼いながらも憧れがありました。そのつながりで超小型PC、UMPC好きになりました。GPD製品をはじめとするUMPCで自分の設定が活きると楽しく、それが学習全般のモチベーションになっています。

## 設定探求の原動力・なぜ知識を深堀したいと考えたのか
私の技術習得の原動力は、「理想のコンピューティング環境を、いかに迅速かつ正確に再構築（レプリケーション）できるか」という探求心にあります。

目的達成のために、例えばArch LinuxのEFISTUBによるブート最適化やi3wm,Xmonadタイルマネージャーの活用による軽量化、NixOSによる宣言的な環境管理、xrandr,udevを用いたハードウェア制御、自宅サーバーやVPSを用いた、Tailscale VPN閉域網接続を活用してのNextcloud,FreshRSSなどのプライベート趣味空間の構築など、低レイヤーからクラウドまで必要な技術を柔軟に取り入れてきました。

特定の抽象化技術に固執することをできるだけ避け、基本的なLAMP環境といったベーシックな構成を重視し、「課題解決、効率化のために最適な道具を選び、最短距離で形にする」スタンスを大切にしています。現在は、点として存在する知識を実務を通じて体系化し、より堅牢なインフラ構築に貢献したいと考えています。

## これらがなぜ実務に生かせると考えるのか
1. システムをコードとして記述し、それを実行することで同一の個人OS環境を再現できるようにすることに情熱を注いできました。
   →　のちに知った「Infrastructure as a Code」の思想と共通すると考え、運用属人化の軽減やシステム再現性の強化の有用性に共感しております。
2. 作成したシステム構成は、稚拙かつシンプルではありますが、できるだけ

## 今後の課題点
1. スクリプティングにおいて、「冪等性」「可搬性」の意識が不十分なこと
2. サーバー環境構築の自動化に未挑戦であること
3. Pythonといった、より自由度が高いプログラミング言語の理解が浅いこと

## Desktop
### Arch Linux
Manjaroライブ環境からinstall.sh実行で自分専用の環境構築を完了するように構成しました。

~~~
chmod +x ./install.sh
./install.sh
~~~

Archベースではありますが、インストールの安定性を確保するため、bootstrapでインストール用環境を一時展開した後、インストール先のディスクを最もシンプルな形なパーティションに再構成します。
その後、インストール→ブートローダー再設定→設定済みi3wm＋個人アプリ導入まで自動化しました。

#### 主に詰まったところ
1. 異なるディストロのライブ環境からのインストールに挑戦してみたところ、bootstrapで一時環境の構築が必要だった。
その際の各種マウントやpacmanの初回処理が慣れておらず、日本語と英語のArchWikiを両方読み、実験用のUMPCの環境を何度もぶっ壊しながら、問題なく動く方法を探った。
2. 設定ファイル書き出しの際の変数展開の有無の制御。
3. ユーザーファイルの権限問題。
i3,i3statusの設定ファイルをほかのシステム設定ファイルと同様に書き込むと、所有者がrootとなり、そのままだとユーザーでログインしたときに設定ファイルが読めない。
これらを見落としており、はまった。

### NixOS
Manjaroライブ環境からinstall.sh実行で自分環境構築を完了するように。

#### 主に詰まったところ
1. Nix言語の書式問題。「結果」から記述していくスタイルは全く初めてで楽しめた。
その反面、関数の選定ミスや記述ミスに加え、コンマの抜けによるコンパイルエラーが多発し、時間を費やした。
また、Archよりも、NixOSはデフォルトの設定のクセが強いと感じることもあり、主観論ではあるが、自分的により「クリーン」と感じる状態にもっていくのがたいへんだったと思う。
3. インストールスクリプトの書き方に詰まった。
関数型OSという慣れない環境特有のはまりポイントだったかもしれない。
```installer.sh```をヒアドキュメントで書き出し、nixパッケージマネージャーをセットアップしたうえで、それを実行。大まかな実行内容としては、```configuration.nix```を編集、```nixos-install```を実行するというシンプルな流れだが、これを見つけるのに手間取った。愚直に公式サイトやドキュメントをよく読み、組み立てればうまくいきやすいことを学んだ。

### Gentoo
設定経験はあるが、コンパイルに時間がかかりすぎるのでトライアンドエラーが十分に回せず、自動化はしていない。

### Linux Mint
使わないプリインパッケージの削除、個人的に必須なアプリのインストール＋PPA追加。
ArchやNixOSでも発生する問題の応急処置的なFixを自動化（GPD製UMPCの画面回転問題、タッチパネル座標関連、Intel系サウンドデバイスのシャットダウン時ポップノイズ抑制など）。
あえて「何もしない」余白を意識し、Mintの強みであるデフォルトの設定を尊重することを心掛けた。

#### 主に詰まったところ
Ubuntuベースで、"out-of-the-box"なディストロということもあり、インストール自体は手間取ることはなかった。
ただ、ArchやGentoo,NixOSが「ゼロから足していく」アプローチを取るのに対し、Mint、MXLinuxなどは自分好みの設定にするにはすでにあるものを「引き算」していくことになるので、まずデフォルトの設定を大まかに理解して、消す、変えるものとそのままにしておくものを選別するのが重要なポイントだと感じた。
画面回転などの設定は、Archなどのものと共通点が多く（どちらもGNU/Linuxベースなので）、部分的にそれを適用するだけで済んだので、元々の知識で何とかなった。

### Windows
使わないUWPアプリの削除、レジストリ操作、手動更新が必要なアプリ（7-Zip,rclone,mpvなど）の起動時更新タスクの追加。

#### 主に詰まったところ
スクリプティングはシェルスクリプトから入ったので、Windows特有の記法に慣れるのに時間はかかった。
ただ、設定に際して複雑な条件分岐などを書くことはあまりなく、基本```reg add```コマンドを羅列するだけで、書いたとしても重複を避けるためにforループを書くぐらいだった。
そのため、自分の希望を叶えるスクリプトを書くだけならば、ハードルとしてはそれほど高くはなかったと感じる。
Mintと同じか、それ以上にデフォルト設定が意味を持ってくる（＝いじり過ぎるとOS自体の動作に致命的）と思うので、ある程度妥協、線引きをするようにはした。
その結果、ほとんど不満なく普段使いできるようにはなった。

## Server
### Nextcloud, FreshRSS
VPSでベーシックなLAMP環境の構築に挑戦した。プライベートで使用するため、VPSに割り当てられた元々のパブリックIPをVPS外部のパケットフィルター、iptablesで遮断、VPN（tailscale)のみで運用することができた。
Nextcloudでは、PHP設定（FPM,OPCACHEまわり）に苦戦したが、公式ドキュメントやエラーメッセージのWeb検索が役立った。

### changedetection.io
主にdocker composeで動かすことが主流のアプリなので、Dockerの知識を再確認することが必要だった。
VPSのスペック限界（月額料金の関係でLightsail,メモリ2GBインスタンス）で、コンテナが知らないうちにハングアップしていることが頻繁に発生したが、定期的にコンテナを立ち上げなおすことで実用上の応急処置をした。
